01 · Product Context

Why Vouch exists

Most QA teams live inside a silent contradiction: test cases written one sprint ago are already slightly wrong, because the AC shifted, the Figma frame changed, or the user story was rewritten — and nobody tracked which downstream artifacts those changes broke. Vouch sits between Jira, Figma, and your test suite, remembers the source-of-truth version each test case was authored against, and refuses to let the link go stale.

In scope for v1

  • Jira Cloud + Figma integration
  • Test case generation from sparse AC + Figma frame
  • Optional meeting transcript as context (no story rewriting)
  • Version-snapshotted link between test case and source
  • Drift detection when AC text or Figma frame changes
  • Stale bug flagging when parent story changes
  • Basic dashboard + in-app notifications

Out of scope for v1

  • User story generation from transcripts (let Rovo / Fathom do it)
  • Automation script generation (Playwright/Cypress) — v2
  • Test execution + reporting — v3
  • Bug auto-filing into Jira — v3
  • Jira Server / Data Center (Cloud only for v1)
  • Non-Figma design tools (Sketch, XD, Penpot)
Primary persona

Maya, QA Lead at a 40-person product startup. 8 years experience, manages 2 mid-level QAs. Lives in Jira and Figma daily. Writes test cases in a mix of TestRail + Google Sheets + her own head. Gets burned twice a quarter by a production bug that slipped past a green suite because the test case no longer matched the feature. Has no budget for enterprise tooling but will expense $30–60/month on a company card without approval.

E1Account & Workspace
3 stories · 7 points
PRD-001User signup and email verification
Must2 pts
As a new user arriving from marketing, I want to create a Vouch account using my work email, so that I can try the product without jumping through hoops.
Acceptance Criteria
  • User can sign up with email + password or Google SSO
  • Verification email is sent within 30 seconds of signup
  • User cannot connect Jira or Figma until email is verified
  • Password must be 10+ characters with 1 digit and 1 symbol
  • Failed signup errors are human-readable (no raw stack traces)
Depends on: None
Stack: Supabase Auth or Clerk
PRD-002Create and name a workspace
Must2 pts
As a newly signed-up user, I want to create a named workspace, so that my Jira/Figma connections and test cases are scoped to the right team and not mixed across clients.
Acceptance Criteria
  • On first login, user is prompted to create a workspace with a name and optional logo
  • Workspace name is editable later from Settings
  • A user can belong to multiple workspaces and switch between them via dropdown
  • Each workspace has independent Jira/Figma credentials, test cases, and billing
Depends on: PRD-001
PRD-003Invite teammates to a workspace
Should3 pts
As a workspace owner, I want to invite my teammates by email, so that a small QA team can collaborate on the same test case library.
Acceptance Criteria
  • Owner can invite up to 5 members in v1 (enforced soft-limit)
  • Invitees receive an email with a one-time magic link valid for 7 days
  • Roles: Owner (1), Admin, Member — Members can review/edit, only Owner/Admin can change integrations
  • Owner can revoke an invitation or remove a member at any time
Depends on: PRD-002
Note: In v1, keep seat model generous — low friction matters more than revenue
E2Jira Integration
4 stories · 16 points
PRD-010Connect Jira Cloud via OAuth
Must3 pts
As a workspace admin, I want to connect my Jira Cloud instance through Atlassian OAuth, so that Vouch can read tickets without me re-entering credentials or pasting API tokens.
Acceptance Criteria
  • Clicking "Connect Jira" opens Atlassian's OAuth 2.0 (3LO) consent screen
  • Scopes requested: read:jira-work, read:jira-user
  • On approval, Vouch stores an encrypted refresh token scoped to the workspace
  • Integrations page shows "Connected" with connected account email and disconnect button
  • Token refresh happens server-side, transparent to the user
  • On revoke or 401, Vouch disables Jira-dependent features and shows "Reconnect" CTA
Depends on: PRD-002
Note: Server-side only — never expose tokens to the browser
PRD-011Select Jira projects and issue types to sync
Must3 pts
As a QA lead, I want to choose which Jira projects and which issue types to pull into Vouch, so that I'm not drowning in archived tickets or non-QA workflows.
Acceptance Criteria
  • After OAuth, user sees a list of all Jira projects with checkboxes
  • User can select issue types per project (Story, Task, Bug, Epic) — defaults to Story + Bug
  • User can scope by status (e.g., "Only pull issues in 'In Progress', 'Ready for QA', 'Done'")
  • Selections are saved and can be edited later
  • Changing selection triggers re-sync within 5 minutes
Depends on: PRD-010
PRD-012Ingest Jira stories and bugs with full content
Must5 pts
As Vouch's backend, I need to pull full ticket content — including description, AC, attachments, linked issues, and custom fields — so that downstream test generation has everything it needs.
Acceptance Criteria
  • Initial sync pulls all selected tickets in paginated batches (100 per call)
  • For each ticket, Vouch stores: summary, description, AC (parsed from common patterns — "Acceptance Criteria:", checkbox lists, "Given/When/Then"), status, labels, links, attachments as URLs
  • Content is stored with a content_hash and synced_at timestamp
  • Incremental re-sync uses updated >= "-10m" JQL to catch recent changes
  • Re-sync runs on a 10-minute cron, configurable to 1-minute for pro users later
  • Figma URLs found in the description are extracted into a figma_refs[] array
Depends on: PRD-011
Stack: Jira REST API v3, node-cron or Supabase Edge Functions + pg_cron
Design note: AC parsing is imperfect in the wild. When AC cannot be clearly extracted, treat the entire description as the spec source — don't force structure that isn't there. The point of Vouch is to tolerate bad inputs.
PRD-013Display synced Jira tickets in a browsable list
Must5 pts
As a QA, I want to browse the list of synced Jira tickets in Vouch, so that I can pick one to generate test cases from.
Acceptance Criteria
  • Paginated table shows: Jira key, title, status, issue type, linked Figma count, test case count, last synced
  • Filter by project, status, issue type, "has test cases" / "needs test cases"
  • Search by text in title or key
  • Click a row to open the ticket detail drawer with full description + AC + Figma links
  • Manual "Re-sync this ticket" button in the drawer
Depends on: PRD-012
E3Figma Integration
4 stories · 18 points
PRD-020Connect Figma account via OAuth
Must3 pts
As a workspace admin, I want to connect Figma with read access, so that Vouch can fetch design frames referenced in Jira tickets.
Acceptance Criteria
  • User authorizes Figma via OAuth 2.0 with file_read scope
  • Refresh token stored encrypted against workspace
  • Only files the user has access to in Figma are readable by Vouch
  • Integrations page shows connected Figma account + "Disconnect"
  • Graceful fallback to Personal Access Token for self-hosted / enterprise users
Depends on: PRD-002
PRD-021Auto-discover Figma files from Jira tickets
Must3 pts
As a QA, I want Vouch to automatically find and fetch Figma files linked from my Jira tickets, so that I don't have to paste links manually for every ticket.
Acceptance Criteria
  • On Jira ticket ingest, scan description + comments for Figma URLs matching figma.com/file/* or figma.com/design/*
  • Extract the file key and optional node-id from the URL
  • Fetch file metadata via Figma API and store against the ticket
  • Users can also manually paste additional Figma links on the ticket detail drawer
Depends on: PRD-012, PRD-020
PRD-022Parse Figma frames into structured test-relevant data
Must8 pts
As Vouch's test generator, I need a structured representation of each Figma frame — inputs, buttons, states, flows — so that I can generate meaningful test cases instead of hallucinating.
Acceptance Criteria
  • For each linked Figma file, fetch the document tree via GET /v1/files/:file_key
  • Walk the node tree and extract: text content, input fields (by naming convention + component type), buttons, icons, states (hover/default/disabled if present as variants), frame-to-frame flow links, image exports
  • Store a canonical JSON representation per frame, with a content hash
  • Extract thumbnail (max 800px) per frame and cache it
  • Support Figma Variants — pull all variants of a component
  • Handle files with 100+ frames without timing out (process async in background)
Depends on: PRD-021
Largest story in E3 — budget 1.5 weeks
Scope warning: Figma nodes are deeply nested and inconsistently named across teams. Aim for 80% extraction accuracy on v1. Perfect parsing is not required — the AI generator will fill gaps.
PRD-023Store versioned snapshots of Figma frames
Must4 pts
As Vouch's drift-detection engine, I need point-in-time snapshots of each Figma frame with a version hash, so that I can compare future versions and detect changes.
Acceptance Criteria
  • Every time a frame is parsed, the structured JSON is saved as a new version row with: frame_id, version_hash, parsed_at, raw_json, thumbnail_url
  • Identical hashes are deduplicated — no new row if nothing changed
  • Figma file's version field from the API is also stored for audit
  • Old versions are retained for at least 90 days for diff rendering
Depends on: PRD-022
E4Test Case Generation
5 stories · 23 points
PRD-030Link a Jira story to one or more Figma frames
Must3 pts
As a QA, I want to confirm or adjust which Figma frames belong to a given Jira ticket, so that the generated test cases are grounded in the right designs.
Acceptance Criteria
  • Ticket drawer shows auto-discovered Figma frames as a checklist
  • User can add, remove, or reorder frames
  • User can paste a Figma URL with a specific node-id to pin to one frame
  • Linking stores the current version_hash of the frame as the baseline
  • Unlinking does not delete the frame snapshot (audit trail)
Depends on: PRD-013, PRD-023
PRD-031Generate test cases from sparse Jira + Figma inputs
Must8 pts
As a QA, I want Vouch to generate a first-draft set of test cases from whatever AC and Figma I have — even if the ticket only says "implement per Figma" — so that I can start with a working outline instead of a blank page.
Acceptance Criteria
  • "Generate test cases" button on the ticket drawer fires an LLM call with: sparse AC (if any), parsed Figma frame JSON, optional transcript text, and a test-generation system prompt
  • Output includes: positive path cases, negative path cases, edge cases (empty state, max length, invalid input), accessibility checks when Figma includes labels/alt text, cross-frame flow cases
  • Each test case has: title, preconditions, steps (numbered), expected result, test type (functional / UI / accessibility / flow), priority (P1-P3)
  • Each test case is linked to one or more Figma frames it was derived from
  • Generation completes in under 60 seconds for a ticket with ≤10 frames
  • Failed generation shows a human-readable error and lets the user retry
Depends on: PRD-030
Stack: Claude Sonnet 4.6 or GPT-4.1 via API
PRD-032Upload meeting transcript as generation context
Should3 pts
As a QA whose PO briefs features verbally, I want to attach a meeting recording or transcript to a Jira ticket, so that test generation knows the decisions that never made it into writing.
Acceptance Criteria
  • Drag-and-drop upload on ticket drawer accepts .txt, .vtt, .srt, .md up to 5 MB
  • For audio/video files: show a message "transcribe externally first — we recommend Otter/Fathom"
  • Uploaded transcript is fed into the test-generation prompt as additional context, not converted into AC or stored as a spec
  • User can delete a transcript at any time; deletion re-generates cases without it on next run
Depends on: PRD-031
Note: Explicitly NOT a story-writing feature. Context only.
PRD-033Review, edit, and approve generated test cases
Must5 pts
As a QA, I want to review each generated test case, edit the ones that need tweaking, and approve them into the library, so that I remain in control of what enters my suite.
Acceptance Criteria
  • Generated cases appear in a review pane with Edit, Delete, Approve, Regenerate actions
  • Inline edit for title, preconditions, steps, expected result, priority
  • Bulk select + bulk approve
  • "Regenerate with feedback" — user types a hint, Vouch regenerates using it
  • Approved cases move to the main test case inventory with status "Active"
  • Unapproved drafts expire after 30 days if untouched
Depends on: PRD-031
PRD-034Persist approved test cases with source bindings
Must4 pts
As Vouch's data layer, I need to persist each approved test case with explicit bindings to the Jira story version and Figma frame version it was authored against, so that drift detection can later check whether those sources have changed.
Acceptance Criteria
  • Each test case has: jira_story_id, jira_content_hash_at_author, figma_frame_ids[], figma_version_hashes_at_author[], status, authored_at, authored_by
  • Editing a test case does not change the "authored against" snapshot — only explicit re-baselining does
  • Test case history is append-only (no hard deletes) — deleted cases are soft-deleted for 90 days
  • Cases can be viewed grouped by Jira story or by Figma file
Depends on: PRD-033
This is the data model that makes drift detection possible
E5Drift Detection
5 stories · 31 points · the differentiator
PRD-040Detect AC/description change on linked Jira stories
Must5 pts
As Vouch's drift engine, I need to notice when the description or AC of a Jira story linked to existing test cases has changed, so that I can flag those cases for review.
Acceptance Criteria
  • On every Jira re-sync, compute content hash of the story description + AC
  • If hash differs from jira_content_hash_at_author on any linked test case, fire a JiraDriftEvent
  • "Material change" heuristic: ignore whitespace-only changes, formatting changes, and comment-only edits — but flag any text addition/removal/substitution
  • Drift event captures: old_content, new_content, diff_summary (one-sentence AI summary of what changed), detected_at
  • Drift events are queryable per test case and per story
Depends on: PRD-012, PRD-034
PRD-041Detect material change on linked Figma frames
Must8 pts
As Vouch's drift engine, I need to notice when a Figma frame linked to existing test cases has materially changed, so that I can flag those cases without spamming users over cosmetic tweaks.
Acceptance Criteria
  • Vouch polls Figma every 6 hours (configurable), OR receives webhook events when available
  • For each frame with linked test cases, compute structural diff against the stored baseline version
  • Material change = text content change, node addition/removal, interaction change, prototype flow change
  • Cosmetic change (color, spacing, typography weight, shadow) → recorded but does NOT trigger "review required" by default
  • User can toggle "include cosmetic changes" per workspace
  • Drift event captures: old_frame_json, new_frame_json, change_type, diff_summary, thumbnail_before, thumbnail_after
Depends on: PRD-023, PRD-034
Core differentiator — this is why someone pays for Vouch
Implementation hint: Use GET /v1/files/:key?version=<id> to fetch old versions. For diff, walk the node tree in parallel and compare by node.id. Budget 1.5 weeks — the "material vs cosmetic" heuristic is the hardest part.
PRD-042Flag affected test cases with "Review Needed" status
Must8 pts
As a QA, I want to see which of my test cases are now at risk because their source changed — and exactly what changed — so that I can update the test cases quickly and confidently.
Acceptance Criteria
  • Any test case linked to a Jira story or Figma frame that has materially changed moves to the "Review Needed" status and shows a yellow "Review Needed" badge in every list view
  • The status applies equally whether the trigger was a Jira user story / AC change (PRD-040) or a Figma frame change (PRD-041)
  • Clicking the badge opens a diff view showing: old vs new AC/story text (highlighted word-level diff), old vs new Figma frame thumbnails side-by-side, AI-generated one-line summary of what changed
  • From the diff view, user can: "Update test case" (triggers auto-update flow — see PRD-044), "Re-baseline without changes" (accept the new source as the new baseline without editing the test case, status → Active), "Mark as irrelevant" (dismiss without baseline change, status → Active), "Edit manually" (opens inline editor)
  • Re-baselining updates jira_content_hash_at_author or figma_version_hashes_at_author to current and returns status to Active
  • History tab on each test case shows all drift events and the action taken for each
Depends on: PRD-040, PRD-041
Status terminology: backend events stay named DriftEvent; user-facing label is always "Review Needed"
PRD-043Review queue — workspace-wide view of everything needing attention
Must5 pts
As a QA lead, I want a single queue view of every test case currently in "Review Needed" status across my workspace, so that I can triage what to fix before the next release.
Acceptance Criteria
  • Review Queue page lists every test case currently in "Review Needed" status with: test case title, source type (Jira story / Figma frame / both), diff summary, detected date, assigned owner, severity
  • Filters: by source type, by owner, by severity, by project, by "last 7 days"
  • Bulk actions: assign to teammate, bulk auto-update (runs PRD-044 across selected), bulk re-baseline (requires confirmation), dismiss
  • Count badge in top nav shows open "Review Needed" items
  • Queue auto-refreshes every 60 seconds while the page is open
Depends on: PRD-042
PRD-044One-click auto-update of test cases flagged for review
Must5 pts
As a QA who just got a "Review Needed" flag on a test case, I want to click a single "Update test case" button and have Vouch regenerate the case against the new source, so that I can approve an already-updated draft in seconds instead of rewriting it myself.
Acceptance Criteria
  • Any test case in "Review Needed" status shows a prominent "Update test case" button in the diff view and on the inventory row
  • Clicking "Update test case" sends the current test case + the new Jira story/AC + the new Figma frame data to the LLM, asking for a targeted, minimal update — not a full rewrite — so steps the user has tuned are preserved where possible
  • The proposed update is shown as a side-by-side diff against the current test case (title, preconditions, steps, expected result, priority all diffed field by field)
  • User can: Accept (replaces test case, re-baselines source hashes, status → Active, logs event), Reject (keeps current test case as-is, re-baselines silently so the user isn't nagged again, status → Active), Edit then accept (opens editor pre-filled with proposed update)
  • Auto-update completes in under 30 seconds per test case
  • Bulk auto-update is available from the Review Queue (PRD-043) and runs up to 10 test cases in parallel
  • If the LLM call fails, show "Couldn't auto-update — edit manually" with the diff still visible; no state change occurs
  • Every auto-update event is logged in the test case History tab with before/after content and who triggered it
Depends on: PRD-031, PRD-042
This is the "magic moment" — the payoff for every user that justifies the subscription
Prompt design note: The update prompt should explicitly instruct the LLM to preserve test case structure and wording where the source change does not require a change, and to show its reasoning in a hidden "change rationale" field the user can expand. This is what makes auto-update feel trustworthy vs. feel like a rewrite.

Should3 pts

Should5 pts

Acceptance Criteria

Should4 pts

05 · Risks & Open Decisions

What could still sink the MVP

Figma parsing accuracy

The node tree Figma returns is inconsistently named across teams. PRD-022 could eat 2 weeks instead of 1.5 if you try to perfect it. Set an 80% extraction target and move on — the LLM will fill the gaps in PRD-031.

Material-vs-cosmetic heuristic

PRD-041's definition of "material change" is the product's most important design call. If too loose, you spam users with noise and they turn off alerts. If too tight, you miss the exact kind of drift users are paying you to catch. Start strict (only text/node/flow changes trigger alerts), add fine-tuning controls later.

LLM cost per generated ticket

PRD-031 can burn $0.50–$2.00 per generation if the Figma frame JSON is large. At 200 free-tier users generating 10 cases each, that's $400–$4,000/month before a single paying customer. Cap free tier to 10 generations/workspace/week.

Jira AC extraction failures

Half your target users have tickets that say "implement per Figma" with no AC at all — that's the pain you're solving. But the other half have AC in bizarre formats (bullet lists of 2 words, screenshots of Confluence pages). Don't over-engineer the parser. Fall back to "treat the whole description as spec" gracefully.

Atlassian Marketplace competition

Xray and Zephyr Scale will ship some version of "AI test generation" within 12 months. Your differentiator is drift detection, not generation. Do not compete on generation quality — compete on the fact that you tell users when their suite goes stale and nobody else does.

Webhook vs poll for Figma

Figma webhooks require team-level config (the user's admin has to approve it). For v1, poll every 6 hours as default. Build webhook support as a "Pro" feature so it also becomes an upgrade trigger.

06 · Working with this spec

How to use this document

Paste each story directly into your tracker — Jira, Linear, Notion, GitHub Issues, whatever. The story IDs (PRD-###) are stable, so linking between them in commits and PRs works cleanly from day one. Story points are rough and meant for sequencing, not billing.

If you're hiring a freelance developer to help, hand them Epic 2 (Jira) and Epic 3 (Figma) first — they're the most mechanical and least opinionated. Keep Epic 5 (drift detection + auto-update) for yourself; that's where the product judgment lives.

When you hit v1 launch, revisit this document before starting v2 planning. By then, real users will have told you which stories you over-built and which ones you under-scoped. Rewrite accordingly.

Launch path — Vercel first, domain later

Deploy on Vercel with the default vouch-app.vercel.app subdomain while recruiting your first 10–20 beta users. Validate: (a) do users link Jira + Figma without help? (b) does the "Review Needed" flag land on the right test cases? (c) does auto-update produce drafts users accept more than 60% of the time? Only buy a domain after those three questions are "yes." Expected Vercel free-tier runway: ~100 active users before you need to upgrade, which is plenty for validation.

Before switching domains, run the trademark + domain availability check on "Vouch" in the classes that matter (Class 042 for SaaS). vouch.io, vouch.dev, vouch.app, vouchqa.com, getvouch.com, tryvouch.com are all viable fallbacks if vouch.com is taken or expensive.
